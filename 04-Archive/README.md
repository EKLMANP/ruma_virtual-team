# 04-Archive - 歷史備份部門

這個資料夾包含 RUMA 龍舟隊專案的歷史資料、舊版本文件和已棄用的內容。

---

## 📁 資料夾結構

```
04-Archive/
├── old-migrations/       # 舊版資料庫遷移記錄
└── deprecated/           # 已棄用的文件
```

---

## 🗄️ Old Migrations - 舊版資料庫遷移

存放歷史的資料庫遷移記錄和備份。

### 用途
- 保留資料庫變更的完整歷史
- 方便追溯問題和進行版本回溯
- 作為參考資料用於未來的遷移設計

### 內容類型
- SQL 遷移腳本
- 資料庫 Schema 快照
- 遷移日誌和記錄
- 回滾腳本

### 注意事項
⚠️ **不要刪除舊的遷移記錄**
- 即使已經不再使用，舊的遷移記錄仍然有參考價值
- 可能需要用於資料恢復或問題追溯
- 保持完整的歷史記錄有助於理解系統演進

---

## 🗂️ Deprecated - 已棄用文件

存放不再使用但需要保留的文件。

### 歸檔原則

#### 應該歸檔的文件
- ✅ 已被新版本取代的功能代碼
- ✅ 不再使用的配置文件
- ✅ 舊版的設計文檔
- ✅ 已完成或取消的實驗性功能
- ✅ 舊的營銷材料和資源

#### 不應該歸檔的文件
- ❌ 仍在使用的代碼或配置
- ❌ 現行的文檔和手冊
- ❌ 敏感資訊（應該安全刪除）
- ❌ 臨時測試文件（直接刪除）

### 歸檔格式

建議在歸檔時添加說明文件：
```
deprecated/
├── feature-name/
│   ├── DEPRECATED.md     # 說明為何棄用、何時棄用、替代方案
│   └── (相關文件)
```

DEPRECATED.md 範例：
```markdown
# 已棄用: [功能名稱]

**棄用日期**: 2026-02-17
**棄用原因**: [簡短說明]
**替代方案**: [新的實作位置或方法]
**相關議題**: [連結到相關的討論或 Issue]
```

---

## 📋 歸檔流程

### 何時歸檔

1. **功能重構** - 舊功能被新實作取代
2. **需求變更** - 業務需求改變，功能不再需要
3. **技術債務清理** - 清理過時的技術實作
4. **定期清理** - 季度或年度的專案整理

### 歸檔步驟

1. **識別** - 確認文件或代碼確實不再使用
2. **記錄** - 創建說明文件記錄棄用原因
3. **移動** - 將文件移至相應的歸檔資料夾
4. **通知** - 通知團隊成員歸檔變更
5. **更新** - 更新相關文檔的參考連結


   ```bash

   ```

## 🧹 清理政策

### 保留期限

- **資料庫遷移**: 永久保留
- **已棄用代碼**: 至少保留 2 年
- **舊版文檔**: 至少保留 1 年
- **測試資料**: 視需要，最短 6 個月

### 定期審查

建議每年審查一次歸檔內容：
- 評估是否仍需保留
- 整理和重新分類
- 刪除過期且無價值的內容
- 更新說明文檔

---

## ⚠️ 重要提醒

### 歸檔 ≠ 刪除

- 歸檔是為了保持主專案的整潔
- 不是為了節省儲存空間
- 歷史資料有其價值，不輕易刪除

### 敏感資訊處理

如果歸檔的文件包含敏感資訊：
- 🔒 在歸檔前先移除敏感資訊
- 🔒 或使用加密保護
- 🔒 考慮是否應該完全刪除而非歸檔

### Git 歷史

- 歸檔的文件仍會保留在 Git 歷史中
- 如需完全移除，需要重寫 Git 歷史（慎用）
- 大多數情況下不需要從 Git 歷史中移除
